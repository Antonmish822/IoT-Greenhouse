**Перечень действий для мейнтейнера проекта по приему Pull Request (PR)**

Как мейнтейнер проекта, ваша задача — обеспечить качество, согласованность и стабильность кодовой базы. Ниже приведены шаги по эффективному ревью и приему Pull Request:

1.  **Первичная проверка и контекст:**
    *   **Ознакомьтесь с описанием PR:** Внимательно прочитайте заголовок и описание PR. Убедитесь, что оно четко объясняет, *что* делает PR, *почему* это необходимо, *как* это было протестировано и на какую Issue (проблему) он ссылается (например, `Closes #123`).
    *   **Проверьте связанные Issue:** Перейдите по ссылке на Issue, чтобы понять весь контекст задачи, обсуждения и принятых решений.

2.  **Автоматизированные проверки (GitHub Actions/CI/CD):**
    *   **Убедитесь, что все CI/CD проверки пройдены:** Это включает линтинг, форматирование, юнит-тесты, интеграционные тесты и успешную сборку Docker-образов. *Никогда не мержите PR, если автоматизированные проверки не пройдены.*
    *   Если какая-либо проверка не пройдена, попросите контрибьютора исправить ошибки.

3.  **Ревью кода:**
    *   **Общий обзор изменений:** Быстро просмотрите измененные файлы, чтобы получить общее представление о масштабе и характере изменений.
    *   **Соответствие стилю и стандартам:**
        *   Проверьте код на соответствие стандартам форматирования (Black) и линтинга (Flake8), даже если CI/CD уже проверяет это. Иногда автоматические инструменты не ловят все нюансы.
        *   Убедитесь, что код соответствует общим принципам и шаблонам проектирования, принятым в проекте.
    *   **Корректность и функциональность:**
        *   Поймите логику изменений. Работает ли код так, как задумано?
        *   Нет ли очевидных ошибок или уязвимостей?
        *   Учитывает ли код граничные случаи?
    *   **Тесты:**
        *   Проверьте, добавлены ли соответствующие тесты для новой функциональности или изменений.
        *   Являются ли тесты достаточно полными? Покрывают ли они важные сценарии, включая граничные случаи?
    *   **Читаемость и поддерживаемость:**
        *   Легко ли читать и понимать код?
        *   Достаточно ли комментариев для сложной логики (где они действительно нужны, а не для очевидных вещей)?
        *   Хорошо ли названы переменные, функции и классы?
    *   **Производительность и эффективность:**
        *   Есть ли потенциальные узкие места в производительности?
        *   Используются ли эффективные алгоритмы и структуры данных?
    *   **Документация:**
        *   Потребовались ли изменения в пользовательской или технической документации? Если да, были ли они внесены и корректны?

4.  **Обратная связь и комментарии:**
    *   Используйте функцию комментариев GitHub для предоставления конструктивной обратной связи. Будьте вежливы и уважительны.
    *   Разделяйте предложения и обязательные изменения.
    *   Если требуется много исправлений, предложите созвониться или обсудить это в другом канале.
    *   Запрашивайте изменения, пока код не будет соответствовать всем стандартам.

5.  **Локальная проверка (при необходимости):**
    *   Для критических или сложных изменений *настоятельно рекомендуется* локально вытянуть ветку PR и протестировать ее в вашей среде разработки. Это особенно полезно для проверки пользовательского опыта, сложных интеграций или воспроизведения ошибок.
    *   **Шаги для локального тестирования PR:**
        1.  **Вытяните ветку PR:**
            *   Добавьте удаленный репозиторий контрибьютора (если еще не добавлен):
                ```bash
                git remote add contributor-username https://github.com/contributor-username/IoT-Greenhouse.git
                ```
                (Замените `contributor-username` на реальное имя пользователя контрибьютора и соответствующий URL его форка).
            *   Вытяните ветку PR:
                ```bash
                git fetch contributor-username pull/PR_ID/head:pr-branch-name
                ```
                (Замените `PR_ID` на номер PR и `pr-branch-name` на осмысленное имя ветки, например, `pr-123-feature-auth`).
            *   Переключитесь на эту ветку:
                ```bash
                git checkout pr-branch-name
                ```
        2.  **Запустите проект:**
            *   Убедитесь, что ваш локальный проект чист (`git status`) и все зависимости установлены.
            *   Остановите любые запущенные экземпляры Docker Compose:
                ```bash
                docker compose down
                ```
            *   Запустите проект с нуля, чтобы убедиться в чистоте сборки:
                ```bash
                docker compose up --build
                ```
            *   Проверьте логи контейнеров на наличие ошибок.
        3.  **Проверьте функциональность:**
            *   Выполните шаги, описанные контрибьютором в секции "Как это было протестировано?" в описании PR.
            *   Проверьте новую функциональность, граничные случаи и убедитесь, что изменения не сломали существующий функционал.
            *   Обратите внимание на пользовательский интерфейс, если PR затрагивает фронтенд.

6.  **Утверждение (Approval):**
    *   После того как все комментарии учтены, изменения внесены и вы удовлетворены качеством кода, одобрите PR.

7.  **Слияние (Merge):**
    *   **Выбор стратегии слияния:** Используйте стратегию слияния, принятую в проекте (например, Squash and Merge для чистой истории, Rebase and Merge для линейной истории, или стандартный Merge Commit). Обычно для feature-веток в GitHub Flow предпочтительнее Squash and Merge или Rebase and Merge.
    *   **Сообщение коммита слияния:** Убедитесь, что сообщение коммита слияния является понятным и информативным, часто ссылаясь на номер Issue.
    *   **Удаление ветки:** После успешного слияния предложите или автоматически удалите ветку PR.

8.  **После слияния:**
    *   Убедитесь, что связанные Issue автоматически закрылись.
    *   Отслеживайте возможные регрессии в основной ветке после развертывания.
    *   Поблагодарите контрибьютора за его вклад.
