# Руководство по внесению вклада

Мы приветствуем ваш вклад в проект "Умная теплица"! Следуя этим рекомендациям, вы поможете нам поддерживать качество кода и эффективно сотрудничать.

## Начало работы

1.  **Форкните репозиторий:** Начните с форка основного репозитория на GitHub.
2.  **Клонируйте свой форк:**
    ```bash
    git clone https://github.com/ВАШ_НИКНЕЙМ/IoT-Greenhouse.git
    cd IoT-Greenhouse
    ```
3.  **Настройте upstream:** Добавьте основной репозиторий как "upstream", чтобы получать обновления:
    ```bash
    git remote add upstream https://github.com/ОСНОВНОЙ_РЕПОЗИТОРИЙ/IoT-Greenhouse.git
    ```
4.  **Установите зависимости и запустите проект:** Следуйте инструкциям в главном [README.md](README.md) для локальной настройки проекта (рекомендуется использовать Docker Compose).

## Стратегия ветвления

Мы используем упрощенную стратегию ветвления **GitHub Flow**.

*   **Основная ветка разработки:** `dev-Antalya` всегда должна быть стабильной и готовой к развертыванию.
*   **Новые функции и исправления:** Вся разработка ведется в отдельных тематических ветках, создаваемых от `dev-Antalya`.
*   **Именование веток:**
    *   `feature/краткое-описание-фичи`: Для разработки новых функций.
    *   `bugfix/номер-бага-или-краткое-описание`: Для исправления ошибок.
    *   `hotfix/краткое-описание-срочного-исправления`: Для срочных исправлений в production.
    *   `docs/краткое-описание-изменений`: Для изменений в документации.

## Сообщения коммитов

Мы придерживаемся [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) для стандартизации сообщений коммитов. Это помогает в автоматической генерации журнала изменений и понимании истории проекта.

Формат сообщения коммита: `<type>(<scope>): <description>`

*   **`<type>` (обязательно):**
    *   `feat`: Новая функция
    *   `fix`: Исправление ошибки
    *   `docs`: Изменения в документации
    *   `style`: Изменения, не влияющие на смысл кода (пробелы, форматирование, отсутствующие точки с запятой и т.д.)
    *   `refactor`: Изменение кода, которое не исправляет ошибку и не добавляет функцию
    *   `perf`: Изменение кода, которое улучшает производительность
    *   `test`: Добавление отсутствующих тестов или исправление существующих тестов
    *   `build`: Изменения, влияющие на систему сборки или внешние зависимости (например, gulp, broccoli, npm)
    *   `ci`: Изменения в файлах и скриптах CI
    *   `chore`: Прочие изменения, которые не изменяют исходный код или тестовые файлы
    *   `revert`: Откат предыдущего коммита
*   **`<scope>` (необязательно):** Указывает место изменения (например, `backend`, `frontend`, `docs`, `auth`, `greenhouse`).
*   **`<description>` (обязательно):** Краткое, императивное описание изменения.

Пример:
```
feat(backend): Добавлена новая конечная точка для управления устройствами

Добавлена функция для создания, чтения, обновления и удаления устройств.
Реализована валидация входных данных с помощью Pydantic.
```

## Стиль кода

Мы используем **Black** для форматирования кода и **Flake8** для линтинга. Перед созданием Pull Request убедитесь, что ваш код соответствует этим стандартам.

*   **Форматирование (Black):**
    ```bash
    # Для бэкенда
    cd backend/ && pip install black && black .
    # Для фронтенда
    cd frontend/ && pip install black && black .
    ```
*   **Линтинг (Flake8):**
    ```bash
    # Для бэкенда
    cd backend/ && pip install flake8 && flake8 .
    # Для фронтенда
    cd frontend/ && pip install flake8 && flake8 .
    ```

## Процесс Pull Request

1.  Создайте новую ветку из `dev-Antalya` для вашей работы.
2.  Сделайте свои изменения.
3.  Убедитесь, что ваш код отформатирован (Black) и прошел линтинг (Flake8).
4.  Напишите или обновите тесты для своих изменений и убедитесь, что все тесты проходят.
5.  Создайте осмысленные сообщения коммитов, следуя соглашениям Conventional Commits.
6.  Отправьте свою ветку в свой форк на GitHub.
7.  Откройте Pull Request из вашей ветки в ветку `dev-Antalya` основного репозитория.
8.  Опишите свои изменения в PR максимально подробно, включая:
    *   **Что изменяет этот PR?** (Описание изменений и их цели)
    *   **Почему эти изменения необходимы?** (Обоснование)
    *   **Как это было протестировано?** (Описание шагов тестирования)
    *   **Какие-либо связанные проблемы GitHub?** (Ссылки на issues)
9.  PR должен быть рассмотрен и одобрен как минимум одним другим разработчиком.

## Сообщение об ошибках и запросы функций

*   **Сообщение об ошибках:** Если вы нашли ошибку, пожалуйста, откройте новую проблему (issue) на GitHub. Включите как можно больше информации: шаги для воспроизведения, ожидаемое поведение, фактическое поведение, версию вашей среды и любые соответствующие журналы.
*   **Предложение функций:** Если у вас есть идея для новой функции, пожалуйста, сначала откройте проблему (issue) для обсуждения. Это поможет убедиться, что функция соответствует целям проекта и избежать дублирования работы.

Благодарим за ваш вклад!
